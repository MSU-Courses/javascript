## Строгий режим - `use strict`

### Что такое строгий режим?

Вы уже знаете, что JavaScript развивался на протяжении многих лет. До 2009 года (ES5) язык развивался абсолютно без проблем обратной совместимости: функции, которые уже были введены, не менялись, а только появлялись новые. Однако в стандарте ES5 были внесены изменения в некоторые функции, уже существующие в языке. Эти изменения сделали язык более строгим, так как до этого многие ошибки в коде не вызывали никаких проблем.

Чтобы не нарушить совместимость с уже существующими проектами, была введена директива `"use strict"`. Используя данную директиву в начале файла, код работает в строгом режиме. То есть, JavaScript начинает вести себя более предсказуемо: _запрещает некоторые небезопасные операции_, выбрасывает ошибки там, где раньше их не было, и тем самым помогает разработчику быстрее находить и исправлять потенциальные проблемы в коде.

Выглядит это следующим образом:

```js
// index.js

'use strict';

// code ...
```

### Какие изменения вносятся строгим режимом?

Вот несколько примеров того, как директива `"use strict"` влияет на выполнение кода JavaScript

1. Запрещает неявное создание глобальных переменных.

   _Без строгого режима:_

   ```javascript
   x = 10; // Это создаёт глобальную переменную "x"
   console.log(x); // 10
   ```

   _Со строгим режимом:_

   ```javascript
   'use strict';

   x = 10; // Ошибка: переменная "x" не была объявлена
   ```

2. Запрещает использование зарезервированных слов в качестве идентификаторов.

   _Без строгого режима:_

   ```javascript
   var let = 10; // Это допустимо, хотя и не рекомендуется
   console.log(let); // 10
   ```

   _Со строгим режимом:_

   ```javascript
   'use strict';

   var let = 10; // Ошибка: "let" - зарезервированное слово
   ```

3. Запрещает дублирование параметров в функциях.

   _Без строгого режима:_

   ```javascript
   function sum(a, a, c) {
     return a + a + c; // Может привести к путанице
   }

   console.log(sum(1, 2, 3)); // 7
   ```

   _Со строгим режимом:_

   ```javascript
   'use strict';

   function sum(a, a, c) {
     // Ошибка: дублирование параметра "a"
     return a + a + c;
   }
   ```

4. Изменяет поведение `this` в функциях.

   _Без строгого режима:_

   ```javascript
   function showThis() {
     console.log(this); // this будет равно глобальному объекту (window в браузере)
   }

   showThis(); // Window {...}
   ```

   _Со строгим режимом:_

   ```javascript
   'use strict';

   function showThis() {
     console.log(this); // this будет равно undefined в строгом режиме в глобальном контексте
   }

   showThis(); // undefined
   ```

### Использовании директивы `use strict` в функциях

Директиву `use strict` можно использовать локально в контексте одной функции, что приведет к применению строгого режима только внутри этой функции, ограничивая его область действия.

```js
function myFunction() {
  'use strict';

  // code ...
}
```

Это позволяет использовать строгий режим в отдельных частях кода, не влияя на остальную часть программы, которая может быть написана в старом стиле.

## Модули

### Предисловие

Как вы считаете, хранить весь код приложения в одном файле - это хорошо или плохо? На этот вопрос нельзя ответить однозначно, потому что всё зависит от размера и сложности программы.

С одной стороны, если приложение совсем небольшое и состоит из нескольких строк кода, то разбивать его на несколько файлов действительно нет особого смысла. Это лишь добавит лишнюю структуру там, где она пока не нужна.

Однако по мере роста кодовой базы хранить весь код в одном файле становится всё менее удачной идеей. Такой подход может заметно усложнить:

- _Поддержку кода_. В большом файле труднее ориентироваться и понимать, где находится нужная логика.
- _Поиск и исправление ошибок_. Чем больше файл, тем сложнее быстро найти источник проблемы.
- _Сопровождение приложения в целом_. Вносить изменения в монолитный файл неудобно, особенно если части программы тесно перемешаны между собой.
- _Тестирование_. Изолировать отдельные части логики и проверять их по отдельности становится труднее.
- _Работу в команде_. Когда несколько разработчиков изменяют один и тот же большой файл, вероятность конфликтов и путаницы возрастает.

Именно поэтому хорошим решением является разделение кода на несколько файлов, каждый из которых отвечает за определённую функциональность. Например, в одном файле может находиться логика работы с данными, в другом - функции для отображения информации, а в третьем - общая логика приложения.

Однако такие файлы не существуют изолированно. Они должны быть связаны между собой, чтобы приложение могло работать как единое целое. Для этого в JavaScript существуют _модули_ - специальный механизм, позволяющий разделять код на отдельные файлы и при этом использовать функции и переменные из одного файла в другом.

На практике существует две основные системы модулей в JavaScript:

- _ECMAScript Modules (ESM)_. Современный стандарт модулей, введённый в ES6 (2015 год) и используемый в браузере и Node.js.
- _CommonJS_. Более старая система модулей, которая исторически широко используется в Node.js.

### Определение модуля в JavaScript

_Модуль в JavaScript_ - это отдельный файл, который может экспортировать функции, переменные и классы, а также импортировать их из других файлов.

Разделяя код на модули, мы получаем возможность использовать функции, переменные и классы из одного файла в другом, а также обеспечиваем обмен функциональностью между различными частями приложения.

### ESM-модули

#### Определение ESM-модуля

Рассмотрим систему модулей ESM (_ECMAScript Modules_). В этой системе для использования функций, переменных или классов из другого файла их необходимо сначала экспортировать в одном файле, а затем импортировать в другом - там, где они действительно нужны.

Важно понимать, что нельзя импортировать функции или переменные, которые не были явно экспортированы. Такой подход вводит чёткие границы между модулями и позволяет контролировать, какие части кода доступны для использования в других файлах.

Для _экспорта_ используется ключевое слово `export`, а для _импорта_ - ключевое слово `import`.

В качестве примера рассмотрим два файла: `add.js`, который содержит функцию `add(a, b)`, и `index.js`, в котором находится основной код программы.

В файле `add.js` мы _экспортируем_ функцию, чтобы сделать её доступной для других модулей:

```js
// add.js

// Экспортируем функцию add для использования в других файлах
export function add(a, b) {
  return a + b;
}
```

Затем в файле `index.js` мы _импортируем_ эту функцию и используем её:

```js
// index.js

// Импортируем функцию add из файла add.js
import { add } from './add.js';

console.log(add(2, 3)); // 5
```

#### Использование ESM-модулей в Node.js

Если вы попробуете запустить эти файлы через node, вы столкнётесь с ошибкой: `SyntaxError: Cannot use import statement outside a module`. Это происходит потому, что по умолчанию Node.js не интерпретирует файлы с расширением .js как ESM-модули.

Существует два основных способа использовать ESM-модули в Node.js:

1. _Использование файлов с расширением .mjs_.

   Достаточно изменить расширение файлов с `.js` на `.mjs`. Например, вместо `index.js` использовать `index.mjs`. В этом случае Node.js автоматически будет воспринимать такие файлы как ESM-модули.

2. _Указание типа модуля в package.json_.

   Файл `package.json` - это конфигурационный файл проекта, который содержит информацию о нём (название, версия, зависимости и т.д.).

   Чтобы включить поддержку ESM, необходимо добавить в него поле `"type": "module"`. Это явно сообщает Node.js, что все `.js` файлы в проекте следует интерпретировать как модули.

   Для этого:
   1. Создайте файл `package.json` в корневой директории проекта.
   2. Добавьте в него следующий код:

   ```json
   {
     "type": "module"
   }
   ```

#### Использование ESM-модулей в браузере

Современные браузеры поддерживают ESM-модули "из коробки". Для их использования достаточно указать тип скрипта как module в HTML-файле. В этом случае браузер будет обрабатывать файл как модуль и позволит использовать `import` и `export`.

```html
<script type="module" src="index.js"></script>
```

Даже если модули подключаются в одном HTML-файле, без использования export и `import` нельзя обращаться к переменным или функциям из одного модуля в другом.

Например,

```html
<script type="module" src="add.js"></script>
<script type="module" src="index.js"></script>
```

Несмотря на то что оба файла подключены к одной странице, в `index.js` нельзя будет использовать функции из `add.js`, если они не были явно экспортированы и импортированы.

Это связано с тем, что каждый модуль имеет _собственную область видимости_. Другими словами, переменные и функции внутри модуля не "видны" за его пределами, если вы явно не разрешите доступ к ним через механизм модулей.

### CommonJS

_Система модулей CommonJS_ - это более старая система модулей, которая исторически широко используется в Node.js. В этой системе для экспорта используется объект `module.exports`, а для импорта - функция `require()`.

Рассмотрим пример:

```js
// add.js

// Экспортируем функции add и div
function add(a, b) {
  return a + b;
}

function div(a, b) {
  return a / b;
}

module.exports = {
  add,
  div,
};
```

Для импорта этих функций в другом файле используется функция `require()`:

```js
// index.js

// Импортируем функции из add.js
const mathModule = require('./add.js');

console.log(mathModule.add(2, 3)); // 5
console.log(mathModule.div(10, 2)); // 5
```

Таким образом, в CommonJS мы сначала экспортируем нужные функции через `module.exports`, а затем получаем доступ к ним в другом файле с помощью `require()`.

## Ключевые слова `export` и `import`

> [!TIP]
>
> Поскольку использование CommonJS становится менее распространённым и является довольно простым, мы не будем на это отвлекаться и сосредоточимся на ESM-модулях, которые являются современным стандартом.

### Что можно экспортировать с помощью `export`?

В JavaScript можно экспортировать практически любые сущности:

- Переменные;
- Константы;
- Функции;
- Классы.

### Именованный импорт и экспорт

#### Именованный экспорт

В случае, когда из модуля требуется экспортировать несколько сущностей, используется именованный экспорт. Для этого перед каждой сущностью указывается ключевое слово `export`.

```js
// player.js

// Экспорт константы
export const player = {
  username: 'Alex',
  hp: 200,
  xp: 10,
};

// Экспорт функции
export function attack() {
  console.log('Player is attacking!');
}

// Экспорт класса
export class Enemy {
  constructor(name) {
    this.name = name;
  }
}

// Экспорт переменной (массива)
export const enemies = [new Enemy('Goblin'), new Enemy('Orc')];
```

Однако на практике часто используется более удобный подход - _группированный экспорт_. Он особенно полезен, когда в файле объявляется много сущностей, и вы хотите отделить их объявление от экспорта.

В этом случае все сущности объявляются как обычно, а затем экспортируются одной инструкцией внизу файла:

```js
// player.js

const player = {
  username: 'Alex',
  hp: 200,
  xp: 10,
};

function attack() {
  console.log('Player is attacking!');
}

class Enemy {
  constructor(name) {
    this.name = name;
  }
}

const enemies = [new Enemy('Goblin'), new Enemy('Orc')];

// Группированный экспорт
export { player, attack, Enemy, enemies };
```

Такой подход делает код более аккуратным: сначала вы описываете логику, а затем явно указываете, что именно хотите "выдать наружу" из модуля и в любой момент можете изменить экспортируемые сущности.

#### Именованный импорт

Чтобы использовать экспортированные сущности в другом файле, их необходимо импортировать. При _именованном импорте_ используется синтаксис с фигурными скобками `{}`, внутри которых перечисляются имена экспортируемых сущностей.

При этом имена должны точно совпадать с теми, которые были указаны при экспорте.

```js
// index.js
import { player, attack, Enemy } from './player.js';
```

Также можно импортировать только те сущности, которые действительно нужны, например только константу `player` и функцию `attack`:

```js
// index.js
import { player, attack } from './player.js';
```

Такой подход позволяет не загружать лишний код и делает зависимости модуля более явными.

### Импорт и экспорт по умолчанию

#### Экспорт по умолчанию

Иногда возникает ситуация, когда из модуля требуется экспортировать только одну сущность. В таком случае используется экспорт по умолчанию с помощью ключевого слова `export default`.

```js
// player.js
const player = {
  username: 'Alex',
  hp: 200,
  xp: 10,
};

export default player;
```

Экспорт по умолчанию предполагает, что в модуле есть одна "главная" сущность, которая и будет использоваться в других файлах. Важно помнить, что в одном модуле может быть _только один экспорт по умолчанию_. Попытка объявить несколько таких экспортов приведёт к ошибке:

```js
const add = (a, b) => a + b;
const div = (a, b) => a / b;

// Ошибка: нельзя экспортировать несколько значений по умолчанию
export default add;
export default div;
```

#### Импорт по умолчанию

При импорте по умолчанию мы получаем конкретную сущность, а не объект с набором экспортов. Поэтому при таком импорте не используются фигурные скобки `{}`.

```js
// main.js
import player from './player.js';
console.log(player.username); // Alex
```

В отличие от именованного импорта, имя переменной при импорте по умолчанию можно выбрать произвольно - оно не обязано совпадать с именем в исходном модуле. Такое поведение связано с тем, что у модуля есть только одна сущность, экспортируемая "по умолчанию", и мы сами определяем, под каким именем будем использовать её в текущем файле.

```js
// main.js
import mainPlayer from './player.js';

console.log(mainPlayer.username); // Alex
```

### Переименование импорта

При импорте сущности можно изменить её имя с помощью ключевого слова `as`. Это может быть полезно, если в текущем файле уже существует переменная с таким же именем или если вы хотите использовать более удобное и понятное название.

```js
// player.js
export const player = {
  username: 'Alex',
  hp: 200,
  xp: 10,
};
```

```js
// index.js
import { player as mainPlayer } from './player.js';

console.log(mainPlayer.username); // Alex
```

В примере выше `player` из модуля `player.js` импортируется под новым именем `mainPlayer`, что помогает избежать конфликтов имён и делает код более читаемым.

Также можно изменить имя сущности непосредственно при экспорте. Это позволяет заранее задать имя, под которым она будет использоваться в других модулях:

```js
// player.js

const player = {
  username: 'Alex',
  hp: 200,
  xp: 10,
};

export { player as mainPlayer };
```

```js
// index.js
import { mainPlayer } from './player.js';

console.log(mainPlayer.username); // Alex
```

В примере выше, при импорте уже не требуется использовать `as`, так как новое имя (`mainPlayer`) было задано на этапе экспорта.

### Импорт всего модуля

Иногда возникает необходимость _импортировать модуль целиком_, не извлекая из него отдельные функции или переменные. Чаще всего это делается в случаях, когда модуль содержит код, который должен выполниться сразу при подключении.

В таком случае используется следующий синтаксис: `import './module.js';`

Рассмотрим пример:

```js
// init.js
function init() {
  console.log('Initializing...');
  // code;
}

init();
```

```js
// index.js

import './init.js';

// code ...
```

При выполнении `index.js` модуль `init.js` будет загружен и выполнен, несмотря на то что из него ничего явно не импортируется.

> [!NOTE]
>
> Такой способ используется относительно редко, так как чаще всего из модулей импортируются конкретные функции, классы или переменные. Однако он _полезен для инициализации приложения_, настройки окружения или подключения побочных эффектов (side effects).

### Импорт всего содержимого модуля

Иногда возникает ситуация, когда в модуле содержится много функций, переменных или классов, и импортировать их по отдельности становится неудобно. В таких случаях можно импортировать всё содержимое модуля сразу в виде одного объекта.

Для этого используется синтаксис `* as`:

```js
import * as moduleName from './module.js';
```

Рассмотрим пример:

```js
// player.js

export const player = {
  username: 'Alex',
  hp: 200,
};

export function attack() {
  console.log('Player attacks!');
}

export function heal() {
  console.log('Player heals!');
}
```

```js
// index.js
import * as playerModule from './player.js';

console.log(playerModule.player.username); // Alex
playerModule.attack(); // Player attacks!
playerModule.heal(); // Player heals!
```

### Когда что использовать?

- Используйте _именованный экспорт_, когда в модуле есть несколько сущностей, которые могут быть полезны в других файлах. Он позволяет импортировать только те части, которые действительно нужны, и делает зависимости более явными.
- Используйте _экспорт по умолчанию_, когда модуль содержит одну основную сущность, которая должна быть доступна при импорте без указания имени.

## Организация кода в модулях

### Когда стоит выносить код в отдельный модуль?

Одним из "философских" вопросов в программировании является вопрос о том, когда стоит выносить код в отдельный модуль. Ответ на него зависит от многих факторов: _размера и сложности приложения, архитектуры проекта, команды разработчиков, используемых библиотек и фреймворков_.

Даже если попытаться ответить на этот вопрос в рамках одной главы, получится скорее эссе, чем чёткая инструкция. Со временем это понимание приходит само - через практику и опыт, вы научитесь "чувствовать".

Вот несколько основных принципов организации кода в JavaScript:

- _Модульность_. Разбивайте код на логические модули, каждый из которых отвечает за определённый функционал или компонент приложения. Это упрощает понимание кода и снижает его сложность.
- _Разделение ответственности_. Каждый модуль или компонент должен отвечать только за одну задачу. Например, модуль, работающий с даннымы, не должен содержать логику отображения интерфейса.
- _Использование функций_. Старайтесь писать небольшие и понятные функции, которые выполняют одну конкретную задачу. Такой код легче читать, тестировать и переиспользовать.
- _Соглашения и стандарты_.Придерживайтесь единых правил оформления кода. Например, можно использовать Airbnb JavaScript Style Guide или Google JavaScript Style Guide - это делает код единообразным и понятным для всей команды.

### Пример организации директорий

Рассмотрим одну из структур директорий, используемых в проектах.

```
project/
│
├── src/
│   ├── index.html
│   ├── index.js
│   ├── styles/
│   │   └── main.css
│   ├── scripts/
│   │   ├── utils/
│   │   │   └── utils.js
│   │   ├── app/
│   │   │   └── app.js
│   │   ├── module1/
│   │   │   └── module1.js
│   │   └── module2/
│   │       └── module2.js
│   └── assets/
│       ├── images/
│       └── fonts/
│
└── package.json
```

В этой структуре:

- `src/`. Директория с исходным кодом (source code). Именно здесь происходит основная разработка приложения.
  - `index.html`. Основной HTML файл вашего проекта.
  - `index.js`. Точка входа в приложение. Здесь обычно происходит инициализация программы и подключение основных модулей.
  - `styles/`. Папка для стилей приложения.
  - `scripts/`. Папка для JavaScript файлов.
    - `utils/`. Директория для утилитарных функций, которые могут использоваться в разных частях приложения.
    - `app/`. Директория для основного приложения, где находится логика работы с данными и взаимодействия между компонентами.
    - `module1/` и `module2/`. Директории для отдельных модулей, которые отвечают за конкретные части функционала приложения.
  - `assets/`. Папка для статических ресурсов, таких как изображения и шрифты.
- `package.json`. Файл, содержащий информацию о вашем проекте и используемых библиотеках.

Это структура - не панацея, а лишь один из примеров организации кода. В реальных проектах структура может быть другой, в зависимости от специфики приложения и предпочтений команды. Но можете начать с такой структуры и адаптировать её под свои нужды по мере роста проекта.
